## 1. 设计哲学

### 1.1 核心定位

**「体验派 + 创作平台」**——手感扎实、视觉出众、对 AI 生成和人工制谱同时友好的音游。

不追求操控机制层面的创新（判定线不旋转、不分裂），而是通过以下三点建立体验差异：

1. **手感打磨到极致**：判定反馈、音效、视觉三通道严格同步
2. **世界观驱动的视觉演出**：角色主题改变整个游玩界面的视觉与听觉风格
3. **AI/UGC 友好的机制复杂度**：Note 类型和轨道设计在表达力与可生成性之间取得平衡

### 1.2 设计原则

| 原则 | 含义 | 反面案例 |
|---|---|---|
| **音乐优先** | 每个 Note 必须对应音乐中的某个声音元素 | 在音乐上堆障碍物而非表达音乐 |
| **可读性不妥协** | 任何时刻玩家都能清晰判断下一步操作 | Note 互相遮挡、与背景撞色 |
| **AI 可生成性** | 所有 Note 类型都有明确的音乐映射规则，AI 可基于规则生成 | 需要人类直觉才能编排的复杂曲线 |
| **渐进复杂度** | 简单谱面只用 Tap + Hold，高难谱面才引入全部类型 | 低难度就塞满所有操作类型 |
| **公平性** | 分数完全由技术决定，不受角色/装备/付费影响 | 卡牌属性影响排行榜得分 |

## 2. 游玩界面布局

### 2.1 屏幕结构

```
┌─────────────────────────────────────┐
│           顶部信息栏                  │
│  [Score: 985,432]   [Combo: 127]    │
│                                      │
│                                      │
│          ◆  ◆     ◆                 │  ← 下落中的 Note
│        ◆      ◆                      │
│      ◆    ◆       ◆    ◆            │
│                                      │
│    ◆   ◆      ◆        ◆           │
│  ◆        ◆       ◆       ◆        │
│                                      │
│══════════════════════════════════════│  ← 判定线
│           判定反馈区域                 │
│  [Great ▲Early]                      │
└─────────────────────────────────────┘
```

### 2.2 默认方向

- **纵向下落**：Note 从屏幕顶部下落至底部判定线
- 默认竖屏（Portrait）——移动端单手持握、双手拇指操作
- 支持横屏（Landscape）模式作为设置选项

### 2.3 安全区域

- 判定线距屏幕底部保留 **15% 屏幕高度** 的反馈区域
- 顶部信息栏不超过 **8% 屏幕高度**，避免遮挡 Note 出现区域
- Note 首次可见位置距顶部至少 **10% 屏幕高度**，给玩家视觉缓冲

## 3. 判定线设计

### 3.1 基本行为

- **位置固定**：水平横贯屏幕底部，不移动、不旋转、不分裂
- **宽度**：100% 屏幕宽度
- **厚度**：约 3-4px（视觉上清晰但不粗笨）

### 3.2 节拍响应

判定线随音乐节拍产生微妙的视觉反馈：

- **脉动（Pulse）**：每个强拍时判定线亮度/粗细产生一次微弱脉冲（±15% 亮度变化），让玩家在视觉外周感知节拍
- **频率**：跟随 BPM，强拍脉冲幅度 > 弱拍

### 3.3 角色主题化

判定线的视觉样式由当前选择的角色决定：

| 角色 | 判定线样式 | 脉动风格 |
|---|---|---|
| **Kael（蛙）** | 锯齿状脉冲线，暗红色 | 强烈闪烁，边缘碎裂粒子 |
| **Lumen（光）** | 柔和流光线，暖黄色 | 温和呼吸感，光晕扩散 |
| **Nyx（夜）** | 虚线星尘轨迹，深蓝色 | 缓慢明灭，星尘飘落 |
| **Aria（咏）** | 声波形曲线，渐变彩色 | 声波振幅跟随音量 |
| **Rei（零）** | 极细直线，灰白色 | 几乎无脉动，偶尔故障闪烁 |
| **Sora（空）** | 裂缝透光线，金色光芒 | 光从裂缝中涌出 |

默认模式（无角色选择）：白色简洁线条，标准脉动。

## 4. Note 类型设计

### 4.1 总览

| Note 类型 | 图标形状 | 操作 | 音乐映射 | 首次出现难度 |
|---|---|---|---|---|
| **Tap** | 菱形 ◆ | 点击 | 鼓点、强拍、清晰节奏点 | Easy |
| **Hold** | 长条 ║ | 按下并保持 | 持续音、弦乐、Pad 音色 | Easy |
| **Flick** | 箭头 △ | 向上快速滑动 | 情绪释放、渐强结尾、甩音 | Normal |
| **Slide** | 圆点链 ●─●─● | 手指沿路径滑过 | 旋律线条、快速音阶、连续音序 | Hard |

### 4.2 Tap Note

**视觉**：
- 菱形（45° 旋转的正方形），尺寸约 48×48dp
- 下落时带微弱拖尾轨迹
- 到达判定线时放大 5% 作为时机提示

**操作**：
- 在 Note 到达判定线时点击对应位置
- 触控判定区域为 Note 中心向外扩展 **±24dp**（适配手指大小）
- 单次触摸即完成，不需要方向

**判定**：
- 以 Note 到达判定线的时刻为基准，按时间偏差判定（见第 6 节）

**AI 生成规则**：
- 映射到 onset（音符起始点）、鼓点、Bass 击打
- 强拍优先放置 Tap
- 最小间隔：同轨道 ≥ 1/8 拍，跨轨道 ≥ 1/16 拍

### 4.3 Hold Note

**视觉**：
- 头部为菱形（与 Tap 相同），尾部为较小的横线标记
- 头尾之间由半透明长条连接，宽度与 Note 等宽
- Hold 期间长条从判定线处逐渐消失（视觉上表达进度）
- 颜色比 Tap 略深，便于区分

**操作**：
- 在头部到达判定线时按下（头判定）
- 保持按压直到尾部到达判定线时松开（尾判定）
- Hold 期间手指可在 X 轴方向微移（±30dp 容差），不会中断

**判定**：
- **头部**：与 Tap 相同的时间窗口判定
- **尾部**：宽松判定（±100ms 内松开均为 Perfect）
- **中途松开**：已覆盖部分按比例得分，但断 Combo
- **总得分** = 头部判定分 × 0.5 + 尾部判定分 × 0.3 + 持续完整性 × 0.2

**AI 生成规则**：
- 映射到持续型音符（弦乐长音、Synth Pad、人声持续段）
- 最短 Hold 时长：1/2 拍
- 高难谱面中可出现 Hold + 另一手 Tap 的组合

### 4.4 Flick Note

**视觉**：
- 向上的箭头形状 △，尺寸略大于 Tap（56×56dp）
- 箭头内有微弱的上升动画（暗示操作方向）
- 边缘带发光效果，与 Tap 明确区分

**操作**：
- 在 Note 到达判定线时，手指从 Note 位置**向上快速滑动**
- 滑动距离 ≥ 40dp 即判定为有效 Flick
- 方向容差：以正上方为基准，**±45°** 范围内均判定为有效（即左上到右上都可以）
- 不要求精确的起始位置——手指在 Note 的 ±30dp 范围内起手均可

**判定**：
- 判定时机基于手指**开始滑动的时刻**（非滑动结束时刻）
- 时间窗口与 Tap 相同

**AI 生成规则**：
- 映射到渐强结尾（Crescendo）、甩音、情绪释放点
- 常放在乐句结尾或 Hold 之后
- 不应出现在高速连击流中间（操作方式切换时间不足会导致挫败）

### 4.5 Slide Note

**视觉**：
- 由一个起始点（实心圆 ●）和若干中继点（半透明圆 ○）组成
- 各点之间由曲线连接，形成滑动路径
- 路径提前显示（比 Note 本身更早出现在屏幕上），给玩家预判时间
- 路径颜色与角色主题色关联

**操作**：
- 手指在起始点到达判定线时按下
- 沿路径滑动，手指经过每个中继点
- 最后一个中继点经过后抬起
- 手指不需要精确贴合曲线——中继点有 **±36dp** 的位置容差

**判定**：
- **起始点**：与 Tap 相同时间窗口
- **中继点**：仅判定"经过/未经过"，时间窗口放宽至 ±100ms，位置容差 ±36dp
- **完成度**：经过的中继点比例影响得分
- **断开重接**：中途抬手后可重新接上后续中继点，但已跳过的点算 Miss

**AI 生成规则**：
- 映射到快速旋律走向、音阶跑动、Arpeggio
- 路径方向应跟随旋律走向（音高上升→路径从左到右或从下到上）
- 最少 3 个节点（起始 + 2 中继），最多 16 个节点
- M1-M2 阶段 AI 仅生成直线/简单曲线路径，复杂路径留给人工制谱

## 5. 轨道系统

### 5.1 基础轨道

- **4 条基础轨道**，均匀分布于屏幕宽度
- 轨道中心 X 坐标：`0.125`、`0.375`、`0.625`、`0.875`（归一化坐标 0.0~1.0）
- 轨道之间无可见分隔线（保持视觉简洁），但 Note 的 X 位置暗示轨道归属

### 5.2 连续坐标系

- Note 的 X 位置使用 **0.0~1.0 连续坐标**，不强制锁定到 4 条轨道
- 低难度谱面：Note 严格放在 4 个轨道中心 → 清晰易读
- 高难度谱面：Note 可出现在任意 X 位置 → 增加读谱难度和表达力
- Slide 路径的中继点可在整个 X 范围内自由定位

### 5.3 AI 生成的轨道使用策略

| 难度 | X 位置策略 | 说明 |
|---|---|---|
| Easy | 严格 4 轨道中心 | 最高可读性 |
| Normal | 4 轨道中心，偶尔相邻轨道间 | 轻微变化 |
| Hard | 连续坐标，但避免极端位置 | 引入位置阅读难度 |
| Expert+ | 全范围连续坐标 | 完整表达力 |

## 6. 判定系统

### 6.1 判定等级与时间窗口

| 判定等级 | 时间窗口 | 得分比例 | 视觉颜色 | Combo |
|---|---|---|---|---|
| **Perfect** | ±45ms | 100% | 金色/角色主题色 | 维持 |
| **Great** | ±90ms | 70% | 白色 | 维持 |
| **Good** | ±130ms | 30% | 灰色 | 维持 |
| **Miss** | >130ms 或未操作 | 0% | 红色（闪烁） | 断开 |

**设计意图**：
- ±45ms 的 Perfect 窗口介于 Arcaea（±50ms Pure）和"明显偏松"之间——大多数节奏感正常的玩家在熟悉后可以稳定拿到 Perfect，但不会松到失去追求精准的意义
- Good 不断 Combo——让手感失误但节奏正确的操作得到基本肯定
- Miss 的红色闪烁清晰告知失误位置

### 6.2 Early / Late 指示

- 非 Perfect 判定时，显示 **Early**（▲）或 **Late**（▼）标记
- 显示位置：**Note 判定位置附近**（不放屏幕角落，因为注意力在轨道上）
- 字号小于判定等级文字，作为辅助信息
- 可在设置中关闭（新手默认关闭，设置中开启）

### 6.3 音频偏移校准

- 首次进入游戏时引导玩家进行**偏移校准**（Offset Calibration）
- 校准方式：播放节拍音，玩家跟随点击 8-16 次，系统计算平均偏差并自动设置
- 设置中提供手动微调（±200ms，步进 5ms）
- 校准值存储在本地，不同设备独立

### 6.4 判定优先级

同一时刻多个 Note 重叠时：
1. 距离手指最近的 Note 优先判定
2. 同距离时，时间上最接近判定线的 Note 优先
3. 已进入 Hold 状态的 Note 不参与新触点的判定竞争

## 7. 分数与评级系统

### 7.1 分数计算

**总分上限：1,000,000**

```
总分 = 精准分 + 连击分

精准分 (满分 900,000):
  每个 Note 精准得分 = (判定得分比 × 900,000) / 总 Note 数
  精准分 = Σ 每个 Note 精准得分

连击分 (满分 100,000):
  连击分 = (最大连击数 / 总 Note 数) × 100,000
```

**Hold Note 特殊计算**：
- Hold 视为 1 个 Note（不拆分头尾），得分按 4.3 节所述的头/尾/持续比例计算

**示例**：
- 一首 400 个 Note 的歌，全部 Perfect + Full Combo = 900,000 + 100,000 = 1,000,000
- 全部 Perfect 但中间断了一次（最大连击 300）= 900,000 + 75,000 = 975,000
- 80% Perfect + 20% Great + Full Combo = (0.8×1.0 + 0.2×0.7) × 900,000 + 100,000 = 826,000 + 100,000 = 926,000

### 7.2 评级

| 等级 | 条件 | 视觉 | 含义 |
|---|---|---|---|
| **Φ** (Phi) | All Perfect (100% Perfect 率) | 特殊全屏动画 + 角色立绘 | 终极成就 |
| **V** | Full Combo (0 Miss, ≥ Good) | 金色徽章 | 硬核里程碑 |
| **S** | ≥ 950,000 | 银色光芒 | 优秀 |
| **A** | ≥ 900,000 | — | 良好 |
| **B** | ≥ 800,000 | — | 中等 |
| **C** | ≥ 700,000 | — | 通过 |
| **F** | < 700,000 | — | 未通过 |

**注**：Φ 和 V 是独立成就标记，与字母评级叠加显示。例如可以同时显示"V + S"（Full Combo 但分数 960,000）。

### 7.3 结算屏幕显示

```
┌──────────────────────────────────┐
│          RESULT                   │
│                                   │
│     ██  Φ  ██     (或 V / S 等)  │
│                                   │
│     Score: 987,654                │
│     Max Combo: 412 / 412          │
│                                   │
│  Perfect: 398  (96.6%)            │
│  Great:     12  ( 2.9%)           │
│  Good:       2  ( 0.5%)           │
│  Miss:       0  ( 0.0%)           │
│                                   │
│  Early: 8    Late: 6              │
│                                   │

└──────────────────────────────────┘
```

## 8. 难度系统

### 8.1 难度等级

**5 个难度标签，数值范围 1-15**：

| 标签 | 数值范围 | 定位 | Note 类型限制 |
|---|---|---|---|
| **Easy** | 1-3 | 零音游经验可通关 | Tap + Hold |
| **Normal** | 4-6 | 有基本节奏感即可 | Tap + Hold + Flick |
| **Hard** | 7-9 | 需要真正技巧 | 全部 4 种 |
| **Expert** | 10-12 | 高手领域 | 全部 4 种 + 连续坐标 |
| **Ethereal** | 13-15 | 传说级 (<5% 玩家能 FC) | 全部 + 高密度 + Gimmick 事件 |

"Ethereal"（以太）呼应游戏名"Aether"，作为最高难度的专属名称。

### 8.2 难度参数参考

| 参数 | Easy (1-3) | Normal (4-6) | Hard (7-9) | Expert (10-12) | Ethereal (13-15) |
|---|---|---|---|---|---|
| 峰值 NPS | 1-2 | 2-4 | 4-8 | 8-14 | 14-20+ |
| Note 类型 | Tap, Hold | +Flick | +Slide | 全部自由 | 全部自由 |
| X 位置 | 4 轨道锁定 | 4 轨道锁定 | 连续坐标 | 全范围 | 全范围 |
| 双手要求 | 单手可完成 | 偶尔双手 | 频繁双手 | 严格双手 | 极限协调 |
| 节奏复杂度 | 整拍/半拍 | +四分音符 | +八分/切分 | +十六分/复合 | 极端切分 |

### 8.3 每首歌的难度配置

- 每首歌**至少提供 2 个难度**（Easy + Hard 或 Normal + Expert）
- 官方曲目尽量覆盖 3-4 个难度
- AI 生成谱面默认生成 3 个难度（Easy / Normal / Hard），高难度由玩家手动制谱或 AI 高级模式生成

## 9. 游玩手感（Game Feel）

### 9.1 判定反馈三通道同步

**所有反馈必须在判定发生的同一帧（≤16ms）内触发**：

| 通道 | 反馈内容 | 时长 |
|---|---|---|
| **视觉** | 判定文字 + 粒子爆发 + Note 消失动画 | 300ms 渐隐 |
| **音效** | Hitsound 播放 | ≤50ms 单次音效 |
| **触觉** | 轻微振动（可选） | 15ms 脉冲 |

### 9.2 视觉反馈细节

**判定文字**：
- 显示在 Note 被判定的位置（非固定位置）
- Perfect：金色，字号较大，轻微上浮动画
- Great：白色，字号中等
- Good：灰色，字号较小
- Miss：红色闪烁，无上浮（强调失败感）

**粒子效果**：
- Perfect：角色主题色粒子向外扩散（8-12 个粒子）
- Great：白色粒子，数量减半
- Good：极少粒子（2-3 个）
- Miss：无粒子，Note 原地碎裂消失

**Combo 显示**：
- 屏幕上方居中，数字随连击增长逐渐变大（最大 150% 基准字号）
- 每达到里程碑（50、100、200、500）触发一次特殊动画
- 断 Combo 时数字震动并快速缩小消失

### 9.3 Hitsound 设计

**默认 Hitsound 组**：
- Tap：清脆的电子敲击声（类似 osu! 默认风格）
- Hold 头部：与 Tap 相同
- Hold 持续中：极低音量的持续 Hum
- Hold 尾部：轻柔的释放音
- Flick：向上的"唰"声（Swoosh）
- Slide 起始：与 Tap 相同
- Slide 中继点：极轻的 Tick 声

**角色 Hitsound 组**（角色选择后替换默认音效）：

| 角色 | Tap 音色 | Flick 音色 | 风格 |
|---|---|---|---|
| Kael | 金属碰撞 | 碎裂声 | Industrial |
| Lumen | 钢琴单音 | 竖琴滑音 | Acoustic |
| Nyx | 低沉水滴 | 气流声 | Ambient |
| Aria | 人声短采样 | 歌声拖尾 | Vocal |
| Rei | 数字脉冲 | 故障噪声 | Glitch |
| Sora | 管弦乐拨弦 | 弦乐颤音 | Orchestral |

### 9.4 Hitsound 音量平衡

- Hitsound 默认音量为音乐音量的 **60%**
- 玩家可在设置中独立调节 Hitsound 音量（0%-100%）
- **Hitsound 与音乐混音后不应互相遮盖**——Hitsound 频段应避开当前歌曲的主旋律频段（由预设的 EQ 处理）

### 9.5 下落速度

- 玩家可自定义 Note 下落速度（影响 Note 出现到到达判定线的时间）
- 范围：**1.0x ~ 6.0x**，步进 0.5
- 默认值：**2.5x**
- 下落速度不影响判定——仅改变 Note 的可见时间
- **关键：下落速度改变的是视觉预判时间，不是音乐速度**

## 10. 世界观与玩法的连接

### 10.1 连接原则

**视觉/听觉层面深度绑定，机制层面保持绝对公平。**

- 不做：角色技能影响判定窗口、角色等级影响分数倍率、付费内容影响游玩能力
- 做：角色改变整个游玩界面的感官体验，让每个角色成为一种"审美滤镜"

### 10.2 角色视觉模板系统

选择角色后，以下视觉元素切换到该角色的主题风格：

| 元素 | 切换内容 | 数据来源 |
|---|---|---|
| 判定线 | 形状、颜色、脉动风格 | `inverse_world_visual.judgment_line_style` |
| Note 皮肤 | 颜色、光效、拖尾轨迹 | 角色主题色 |
| 粒子效果 | Perfect 粒子形状/颜色/运动方式 | `inverse_world_visual.particle_style` |
| 背景 | 底色、动态粒子、氛围光 | `inverse_world_visual.background_mood` |
| Hitsound | 全套音效替换 | 角色 Hitsound 组 |

### 10.3 动态背景系统

背景不是静态图片，而是对音乐实时响应的粒子/着色器系统：

- **低频能量**（Bass）→ 背景色彩脉动强度
- **中频能量**（Melody）→ 粒子运动速度
- **高频能量**（Hi-hat/Cymbal）→ 粒子闪烁频率
- 叠加角色主题色板，形成"同一首歌、不同角色、不同视觉体验"

### 10.4 谱面 Gimmick 事件

RACF 格式中的 `events` 字段支持纯视觉事件：

| 事件类型 | 效果 | 对判定的影响 |
|---|---|---|
| `cameraShake` | 屏幕微震 | 无 |
| `colorFlash` | 全屏闪色 | 无 |
| `inkBurst` | 墨水扩散效果 | 无 |
| `backgroundShift` | 背景渐变切换 | 无 |
| `speedChange` | Note 下落速度变化 | 无（仅视觉加速/减速） |
| `laneNarrow` | 轨道视觉收窄 | 无（判定区域不变） |

**关键约束**：所有 Gimmick 事件**绝不改变判定逻辑**。`speedChange` 改变的是视觉下落速度，Note 到达判定线的时机与音乐严格同步不变。

## 11. 输入处理

### 11.1 多点触控

- 支持最多 **10 点同时触控**（覆盖所有合理的多指操作）
- 每个触控点独立追踪，不互相干扰
- 新触点不影响已有 Hold 的判定状态

### 11.2 触控区域

- 每个 Note 的可触发区域为以 Note 中心为圆心、**半径 36dp** 的圆
- 相邻 Note 触发区域可重叠——此时取距离触点最近的 Note
- 判定线附近（±40dp 垂直范围）的触控才参与 Note 判定，屏幕其他区域的触摸被忽略（避免误触）

### 11.3 输入延迟补偿

- 触控事件时间戳应使用系统事件时间（非帧处理时间）
- Godot 中通过 `InputEventScreenTouch.timestamp` 或自定义高精度计时获取
- 补偿公式参考（Godot 4.3+）：

```
实际判定时间 = 触控事件时间 - 用户设置的偏移量
Note 目标时间 = Beat 转换为秒 + 音频偏移
判定差值 = 实际判定时间 - Note 目标时间
```
